  1. Controllers

    1. Sensors and Actuators

      2. Sensors and actuators are based on the Johnny-Five API. Expanders may be used along with a compatible firmware for the Arduino such as ConfigurableFirmata. Systems on a chip(SOC) such as the RaspberryPi that can run the Node.js environment can be used to run the full controller software without an accompanying WiFi connection to such an environment.

      IO pin expanders will be used to share the address space from one SOC so that all the sensors and actuators can connect to only one board. The sensors-actuator pairs are as following:

      • Sensor: digital thermometer at setting 37. Actuator: a heating pad with either boolean or variable temperature control.
      • Sensor: pH analog at setting 7. Actuator: a servomotor to pump basic solution.
      • Sensor: O2 analog at setting 4. Actuator: a servomotor to manage O2 flow.
      • Sensor: digital light at setting .4. Actuator: stop bioreactor and alarm.

      Each sensor’s value is updated on a defined global frequency. That value will be pushed to the browser clients’ time-series charts automatically.

    2. Control Loops

      3. In a different loop and the same frequency as the sensor updating, the sensor’s values will be compared to their set goal value. Based on that comparison, each actuator is put into its relevant state: positive or negative.

    3. Alarm

      4. A special control loop is an end-of-cycle alarm. This alarm triggers all other actuators to shut off the bioreactor and sound an alarm. There is no positive action for this sensor because it is passively listening to the density of the bacteria in the system.

    4. Real-time Controllers

      5. The real-time control system will rely on a shared plan of execution using NTP synchronization with the command server. The only information that will be passed between this controller and the aggregation and command server are incoming execution plans, synced time, and outbound current sensor values.

  2. Aggregation and Command

    3. The aggregation and command server will In a typical scenario, there will only be one bioreactor, but there could be multiple. This server will handle communication of plans to the controllers and provide the facilities of synchronization of the controller central clock. It also broadcasts the automated updating between the real-time controllers and browser client through a single-user interface. There can be multiple browser clients using the interface at one time, but they will all be updated with the same information and share the same hardware.

  3. Browser Clients

    4. The browser clients will chart time-series sensor data and group it by bioreactor instance. They will also allow to set goal settings and plans for each individual bioreactor. Browser-specific tools will involve CSV export for data and plans, as well as diff tools for plans.
